================================================================================
TASK-004: Fix RawExpr SQL injection: return error instead of interpolating verbatim
Date: 2026-05-27T00:00:00Z
================================================================================

Summary: Closed the highest-severity SQL injection vector in the translator.
The `case *cypher.RawExpr` branch in `exprToSQL` previously returned `e.Text`
verbatim into the generated SQL string. It now gates on `cypher.IsIdentifier(e.Text)`:
safe identifiers (letters/digits/underscores only) are passed through; all other
values return a descriptive error. To enable cross-package use, `isIdentifier` in
`cypher/planner.go` was exported to `IsIdentifier` and all callers in
`cypher/planner.go` and `cypher/parser.go` were updated accordingly.

Files Changed:
- sql/translator.go (case *cypher.RawExpr: gate on IsIdentifier, return error for unsafe inputs)
- cypher/planner.go (export isIdentifier → IsIdentifier; update all internal callers; improve doc comment)
- cypher/parser.go (update sole caller to IsIdentifier)

Key Decisions:
- Exported `isIdentifier` rather than duplicating the logic in the sql package — keeps a single source of truth for the allowlist
- Simple identifiers (variable names resolved via RawExpr) remain valid passthroughs; complex expressions (function calls, arithmetic, etc.) now error cleanly
- No existing tests exercise the RawExpr passthrough path in translator, so no test changes were needed

TASK-005: Validate property key names before JSON path construction
Date: 2026-05-27T00:00:00Z
================================================================================

Summary:
Added `validatePropKey(key string) error` to `sql/translator.go`. Validation
gates on `cypher.IsIdentifier`, rejecting keys that contain injection characters
(`]`, `[`, `"`, null bytes, etc.). Applied at all JSON path construction sites:
MATCH node/rel/var-length prop loops, exprToSQL PropExpr, ExistsPropExpr,
GROUP BY PropExpr, translateSetProp, translateSetMerge, translateRemoveProp,
translateMerge, and buildPropsJSON (CREATE path). The `$` planner sentinel key
is explicitly exempted. Key validation in translateSetProp was reordered to
occur before value expression evaluation.

Files Changed:
- sql/translator.go (validatePropKey + calls at all JSON path sites; $ sentinel exemption; translateSetProp reorder; buildPropsJSON coverage)
- sql/translator_test.go (3 new tests: ClosingBracket, OpeningBracket, NullByte rejection)

Key Decisions:
- $ is exempted in validatePropKey itself (not at call sites) so the exemption is documented once and cannot be missed by new call sites
- translateSetProp validates key before building value expression to surface the security-relevant error first
- buildPropsJSON now validates keys, closing the CREATE/MERGE injection path

Date: 2026-05-27T00:00:00Z
================================================================================

Summary:
Added `WithMaxPathHops(n int)` option to both `options.go` (DB-level) and

with documentation; error message no longer hardcodes package name.

Files Changed:
- options.go (WithMaxPathHops option; n<=0 is no-op with docs)
- driver.go (maxPathHops field on DB; propagation through runQuery/BeginTx)
- tx.go (maxPathHops field; passed to runQueryTx)
- sql/translator.go (TranslatorOption/WithMaxPathHops; hopCap enforcement; generic error message)
- sql/translator_test.go (tests: explicit cap rejection, WithMaxPathHops raises cap, unbounded still clamped)

Key Decisions:
- n<=0 is silently treated as "use default 15" (documented); returning an error at Open time for n<=0 would be too strict for zero-value struct initialization patterns
- Error message says "use WithMaxPathHops" without package prefix to avoid stale cross-package references
